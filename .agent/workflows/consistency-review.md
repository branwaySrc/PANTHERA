# 주석 로그 기반 정합성 검토 프로세스 (Consistency Review Workflow)

이 워크플로우는 2026-02-23 이후 모든 작업의 `implementation_plan.md` 작성 시 필수적으로 적용되는 정합성 검토 절차입니다.

## 1. 개요

작업하고자 하는 코드 내의 JSDoc, 리팩토링 제안,
경고 및 기타 주석들을 사용자의 현재 요청과 비판적으로 대조하여
설계의 원천(Source of Truth)을 보호합니다.

## 2. 검토 수행 단계 (Implementation Plan 작성 전)

1. **코드 분석**: 수정 대상 파일의 상단 JSDoc(@description, @refactor-suggestions 등)과 로직 내 주석을 모두 읽습니다.
2. **요청 대조**: 사용자 명령이 기존에 명시된 로직의 의도나 데이터 구조와 충돌하는지 확인합니다.
3. **위험 평가**: 이번 수정이 과거에 명시된 '주의점'이나 '추후 고려사항'을 훼손하거나 누락시키는지 평가합니다.

## 3. Implementation Plan 출력 형식 (필수 섹션)

```markdown
# [Goal Description]

## 검토 결과 보고 (Consistency Review Report)

- **명령-명세 일치 여부**: [일치 / 불일치 / 질문 필요]
- **상세 내용**: (한글 작성)
- **주의/경고**: (과거 주석 기반으로 이번 작업 시 발생할 수 있는 위험 요소 명시)

## Proposed Changes

...
```

## 4. 대응 원칙

- **비판적 검토**: 사용자의 명령이 기존 설계 의도와 다를 경우 맹목적으로 따르지 않고,
  주석된 근거를 바탕으로 질문을 던집니다. (예: "현재 주석에는 A로 명시되어 있으나 명령은 B입니다. 어느 쪽이 맞습니까?")
- **정합성 우선**: 주석 로그를 설계의 핵심 가이드라인으로 삼아 정합성을 유지합니다.
