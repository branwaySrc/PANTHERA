# Supabase 스키마 변동성 및 타입 정합성 검토 워크플로우

##1. Supabase 변경 분석 단계:

- 사용자가 스키마(schema-definition.md)를 수정하거나 새로운 테이블/컬럼 추가를 요청할 경우, '이전 스키마 명세'와 '현재 요청'을 즉시 대조할 것.
- 변경된 부분이 기존의 데이터 흐름(예: 프로젝트-파일 관계)을 파악하여 파생되는 영향도(Impact Analysis)를 나에게 먼저 보고할 것.

##2. 업데이트 주석 및 이력 관리:

- 추가 또는 수정된 컬럼/테이블과 관련된 모든 컴포넌트, 훅, 타입 파일에 [UPDATE] 주석을 달 것.
- 주석 형식: // [UPDATE: 2026-02-24] 사유: 파일 노드 정렬 순서(sort_order) 컬럼 추가에 따른 정렬 로직 반영
- 만약 기존 로직과 충돌이 예상된다면, 수정을 진행하기 전 나에게 "이전 구조는 A였으나 요청은 B입니다. 기존 로직을 유지할까요, 아니면 마이그레이션할까요?"라고 반드시 질문할 것.

##3. Implementation Plan 반영 사항:

- [Supabase 정합성 검토]: 신규 추가된 DB 필드가 기존의 'database.types.ts' 및 Zustand Store와 어떻게 동기화되는지 구체적인 계획을 포함할 것.
- [마이그레이션 위험 평가]: 기존에 저장된 Seed 데이터나 사용자 데이터가 새로운 스키마에서 깨질 위험이 없는지 체크하여 리포트할 것.

##4. 대응 원칙:

- "과거의 정합성을 파괴하지 않는 확장"을 원칙으로 하며, 모든 DB 관련 수정은 경로, src/db/schema-definition.md를 최우선 기준으로 삼음.
